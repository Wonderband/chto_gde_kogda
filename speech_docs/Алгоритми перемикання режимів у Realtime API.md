Перемикання між режимами — це ключовий технічний момент, який забезпечує «магію» живого ведучого. Відповідно до джерел, цей процес не відбувається миттєво сам по собі; його ініціює ваш **додаток (App)**, який виступає в ролі «диригента» 1, 2\.  
Ось детальний план того, як саме ваш додаток зрозуміє, що настав момент для «Гонгу» та переходу в монолог:

### 1\. Механізм відстеження тексту (Transcript Monitoring)

Найкращий спосіб реалізувати це — використовувати подію **response.audio\_transcript.delta** 3, 4\.

1. **Як це працює:** Модель генерує аудіо та текст одночасно, але робить це **швидше за реальний час** 5, 6\. Це означає, що ваш додаток отримає текстовий фрагмент «А тепер — увага, запитання\!» за кілька сотень мілісекунд до того, як користувач почує ці слова з динаміків 6, 7\.  
2. **Тригер:** Ваш додаток аналізує вхідний потік текстових дельт. Щойно він бачить ключові слова («увага», «запитання»), він миттєво запускає дві команди:  
3. **session.update**: встановлює turn\_detection: null (вимкнення VAD), що робить модель «глухою» до переривань 8, 9\.  
4. **Відтворення звуку:** Програє файл із записом «Гонгу» поверх аудіо-потоку.

### 2\. Використання інструментів (Tool Calling) як жорсткого тригера

Для більшої надійності ви можете надати моделі інструмент (функцію) під назвою **trigger\_question\_mode** 10, 11\.

* **Інструкція для ШІ:** У системному промпті ви вказуєте: «Коли закінчиш small talk і будеш готовий зачитати запитання, спершу виклич функцію trigger\_question\_mode, а потім одразу кажи: "Увага, запитання\!"».  
* **Реакція додатка:** Щойно додаток отримує подію про виклик цієї функції (що відбувається ще *до* того, як модель почала говорити фразу), він:  
* Надсилає session.update для вимкнення режиму діалогу 2, 8\.  
* Дає команду фронтенду бути готовим до «Гонгу».

### 3\. Технічна черговість «перемикання»

Щоб уникнути ситуації, коли гравці перебивають модератора на півслові «Увага...», послідовність має бути такою:

1. **App** бачить у дельтах транскрипту або через виклик інструменту, що ШІ переходить до запитання 2, 3\.  
2. **App** негайно надсилає input\_audio\_buffer.clear (щоб видалити випадкові шуми гравців, які могли потрапити в буфер в останню секунду) 9, 12\.  
3. **App** оновлює сесію на turn\_detection: null 8\.  
4. **App** на стороні клієнта (в браузері) програмно «мутить» мікрофон користувача, щоб жоден звук більше не йшов на сервер 13\.  
5. Тепер модератор у повній тиші та без ризику бути перерваним зачитує текст запитання.

### Чому це спрацює?

Завдяки тому, що Realtime API видає дані **чанкам (chunks)** і робить це з випередженням реального мовлення, ваш додаток завжди має «фору» в часі 5, 6\. Ви встигнете вимкнути прослуховування саме в той момент, коли модель почне вимовляти вашу фразу-тригер.  
**Ваша роль як розробника:** написати логіку на стороні сервера (або клієнта, якщо ви використовуєте WebRTC), яка слухає подію response.audio\_transcript.delta і при збігу з шаблоном «Увага...» виконує зміну конфігурації сесії 3, 14\.
